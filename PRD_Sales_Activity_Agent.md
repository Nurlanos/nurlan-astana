# PRD: AI Sales Activity Logger
**Продукт:** AI-агент для записи активности продажника  
**Компания:** Казахтелеком  
**Формат:** Хакатон MVP  
**Версия:** 1.0 (Draft)  
**Дата:** 2026-04-20  
**Статус:** На согласовании  

---

## 1. Контекст и проблема

Продажники Казахтелекома тратят от 30 до 40 минут в день на ручной ввод активностей в CRM — звонки, встречи, отправленные коммерческие предложения, смена этапов сделки. Данные вносятся нерегулярно и частично теряются. Руководитель не видит картину активности команды в реальном времени.

**Цель продукта:** снизить время ввода активностей до менее 1 минуты за событие за счёт NLP-парсинга свободного текста и автоматической записи через оркестратор Camunda 7.

---

## 2. Целевые роли (Personas)

### Продажник
- Фиксирует активности через чат-бот текстом в свободной форме
- Хочет тратить минимум времени на административную работу
- Нуждается в личной сводке по своей активности за день/неделю

### Менеджер / Супервайзер
- Видит дашборд активности всей команды в реальном времени
- Получает уведомление если у кого-то из команды нет активностей за день
- Принимает управленческие решения на основе данных

---

## 3. User Stories

### Продажник

**US-01.** Как продажник, я хочу написать в чат свободным текстом что произошло (звонок, встреча, отправил КП), чтобы агент сам структурировал и записал активность без заполнения форм.

**US-02.** Как продажник, я хочу видеть подтверждение записи сразу в чате, чтобы знать что активность зафиксирована корректно.

**US-03.** Как продажник, я хочу видеть свою сводку активностей за день и неделю, чтобы отслеживать собственный прогресс по плану.

### Менеджер

**US-04.** Как менеджер, я хочу видеть дашборд активностей всей команды в реальном времени, чтобы понимать кто работает активно, а кто отстаёт.

**US-05.** Как менеджер, я хочу получать уведомление если у продажника нет активностей за рабочий день, чтобы вовремя подключиться.

## 5. Архитектура системы

### Стек технологий

| Компонент | Решение |
|---|---|
| Оркестратор | Camunda 7 |
| NLP / LLM слой | Claude API (Sonnet) |
| Бэкенд | Mock-сервис (синтетические данные) |
| Канал ввода | Чат-бот (текстовый ввод) |
| Хранилище | Mock CRM (REST API) |
| Дашборд | Веб-интерфейс (2 роли) |

### Архитектурный поток (Happy Path)

```
[Продажник] → текст в чат
      ↓
[Чат-бот] → передаёт сообщение
      ↓
[Camunda 7: Start Event] — триггер новой активности
      ↓
[Service Task: LLM NLP] — парсинг Intent + Entities
      ↓
[XOR Gateway] — определение типа активности
      ├── Звонок → [Service Task: запись звонка]
      ├── Встреча → [Service Task: запись встречи]
      ├── КП/письмо → [Service Task: запись КП]
      └── Этап сделки → [Service Task: обновление этапа]
            ↓
[Service Task: Mock CRM] — сохранение записи
      ↓
[Service Task: Follow-up] — создание задачи (если нужна)
      ↓
[End Event] — подтверждение в чат продажнику
```

### Извлекаемые сущности (NLP)

При парсинге свободного текста агент извлекает следующие сущности:

- **тип активности** — звонок, встреча, КП, этап сделки
- **клиент** — имя или название компании
- **дата и время** — явные или относительные ("сегодня", "вчера")
- **длительность** — для звонков и встреч
- **итог** — результат / договорённость
- **следующий шаг** — follow-up действие

## 7. Дашборд активности

### Уровень продажника (личный)
- Счётчики активностей за день и неделю по типам
- Статус выполнения плана активности
- Список последних записей с возможностью просмотра деталей
- Открытые follow-up задачи

### Уровень менеджера (командный)
- Сводная таблица активностей по каждому продажнику
- Цветовая индикация: зелёный — выполнен план, жёлтый — риск, красный — нет активностей
- Фильтры по дате, типу активности, сотруднику
- Уведомления о неактивных продажниках

---

## 8. Нефункциональные требования

- **Время ответа агента:** не более 5 секунд от отправки сообщения до подтверждения
- **Точность NLP-парсинга:** не менее 85% корректно распознанных сущностей на тестовом наборе
- **Доступность:** 99% в рабочие часы (09:00–18:00 по Алматы)
- **Язык ввода:** русский и казахский (приоритет — русский для MVP)
- **Безопасность:** аутентификация пользователей, изоляция данных по ролям

## 9. Дорожная карта итераций

Подробная таблица итераций — см. файл `Roadmap_iterations.csv`.

### Итерация 1 — Хакатон (MVP)
Текстовый ввод → NLP-парсинг → Camunda BPMN → Mock CRM → базовый дашборд для двух ролей.

### Итерация 2 — Пост-хакатон
Голосовой ввод (Speech-to-Text), валидация и уточнение у продажника, follow-up задачи и напоминания, интеграция с реальной CRM.

### Итерация 3 — Продуктив
AI-рекомендации по сделкам, обнаружение аномалий в активности, прогноз выполнения плана, интеграция с телефонией.

---

## 10. Ограничения и допущения

- Для хакатона используются только синтетические данные через Mock-сервис
- Голосовой ввод выносится за рамки MVP
- Интеграция с реальной CRM Казахтелекома — вне скоупа хакатона
- Предполагается, что у каждого продажника есть уникальный user_id для идентификации

---

## 11. Критерии успеха MVP

- Агент корректно распознаёт тип активности из свободного текста в 85%+ случаев
- Запись активности в Mock CRM занимает менее 5 секунд
- Дашборд отображает данные для обеих ролей без ошибок
- Демонстрация полного сценария: ввод → запись → отображение на дашборде
